# 数学建模技能 - Math Modeling

本技能为数学建模竞赛和项目提供结构化的三阶段工作流程：建模分析、代码实现、论文撰写。确保建模、编程、论文撰写三个环节紧密衔接，产出高质量的数学建模成果。

## 技能概述

数学建模是利用数学方法解决实际问题的过程，核心流程包括：

1. **问题分析**：理解题意，选择合适的数学模型
2. **模型建立**：将实际问题转化为数学表达式
3. **模型求解**：通过编程实现模型求解
4. **结果分析**：对结果进行解释和可视化
5. **论文撰写**：将完整过程整理成学术论文

本技能采用三阶段协作模式，各阶段按顺序依次执行。

## 产出标准（硬性规则）

### 论文结构

论文按 **每问独立** 的结构组织，每个问题包含完整的方法对比、模型建立、求解、结果分析：

```
摘要（含关键词）
目录（按竞赛要求决定是否保留）
一、问题重述
二、问题分析（整体思路 + 各问关联分析）
三、模型假设
四、符号说明
五、问题一的模型建立与求解
    5.1 方法对比与选择（>= 2 种方法）
    5.2 模型建立
    5.3 求解过程
    5.4 结果与分析
六、问题二的模型建立与求解（同上结构）
七、问题三...
八、模型评价与推广
参考文献
附录（完整代码 + 运行数据表格）
```

### 图表标准

| 项目 | 要求 |
|------|------|
| 每问图片数 | **>= 4 张**（越多越好） |
| 每问表格数 | 0 - 2 个（原始数据 / 结果对比 / 方法对比） |
| 每张图分析文字 | >= 100 字 |
| 图片格式 | PDF（矢量）或 300dpi PNG |
| 图片命名 | `q{问题号}_fig{序号}.pdf` |

### 方法对比要求

每一问必须先列出 **>= 2 种候选方法**，用对比表格（适用性/精度/效率/可解释性）展示，正文中对每种方法简要描述原理（各 100-200 字），然后明确选择理由后再展开详细求解。

### 页数控制

| 部分 | 页数限制 |
|------|--------|
| 正文（摘要到评价与推广） | **<= 25 页** |
| 参考文献 | 不计入 |
| 附录（代码 + 运行数据） | 不计入 |

### 输出格式

LaTeX 项目（分多个 `.tex` 文件，编译为单个 PDF）：
- 模板位置：`references/latex-template/`
- 排版指南：`references/latex-guide.md`
- 编译方式：`latexmk -xelatex main.tex`

## 工作流程

### 第一阶段：建模分析

**⚠️ 执行前提（必须最先执行）**：

在开始建模分析之前，**必须先读取**以下文档：
- `references/roles/建模手说明.md` - 详细的建模分析工作流程、模型选择原则
- `references/problem-decomposition.md` - 问题分解启发式指南（变量识别、约束提取、目标函数构建、决策树）
- `assets/算法选择决策指南.md` - Head-to-head 算法对比、反模式清单、组合方案

**执行任务**：分析题目，确定数学模型，产出分析文档。

执行步骤：

- 使用 pdf skill 读取题目 PDF，**立即将原题输出为 markdown 文件**（如 `C题.pdf` -> `C题.md`），保留原始格式、表格和图片引用
- 阅读并理解题目要求
- **分析各问之间的依赖关系**（依赖/扩展/独立），绘制问题依赖图
- 按 `references/problem-decomposition.md` 中的协议提取变量、约束、目标函数
- 为每道题确定 **>= 2 种候选方法**，参考 `assets/算法选择决策指南.md` 做 head-to-head 对比后选定最优方法
- 记录模型对应的公式、论文引用等信息
- 创建术语表格保持术语一致性

**重要约束**：本阶段不编写代码，仅产出分析文档供后续阶段使用。

**产出文件**：

- `X题.md` - 原题的 markdown 格式副本（读题后立即生成）
- `题目分析报告.md` - 包含问题分析、问题依赖图、多方法对比、模型选择、公式推导、参考文献
- `术语表格.md` - 中英文术语对照表

### 第二阶段：代码实现

**⚠️ 执行前提（必须最先执行）**：

在开始代码实现之前，**必须先读取**角色指导文档：
- `references/roles/编程手说明.md` - 详细的代码实现工作流程、代码规范、可视化要求

**执行任务**：根据题目原文和建模阶段产出的文档，编写解题代码并生成结果。

执行步骤：

- 确认用户指定的编程语言（Python、MATLAB等）
- 按每道题目分开编写结构清晰的代码
- 运行代码并将结果写入表格（保持表格原有结构）
- **每问生成 >= 4 张**符合SCI/Nature风格的精美图表，保存为 PDF 或 300dpi PNG
- **结果验证**：对每问结果进行量级检查、边界测试、基线对比
- 创建README文档说明项目

**产出文件**：

- `问题X_求解.py`（或其他语言）- 各题目的解题代码
- `results/` - 计算结果（CSV + TXT）
- `figures/` - 可视化图表（命名：`q{问题号}_fig{序号}.pdf`）
- `README.md` - 项目说明文档

### 第三阶段：论文撰写

**⚠️ 执行前提（必须最先执行）**：

在开始论文撰写之前，**必须先读取**以下文档：
- `references/roles/论文手说明.md` - 详细的论文撰写工作流程、撰写规范
- `references/latex-guide.md` - LaTeX 排版指南（公式、图表、引用、附录格式）

**执行任务**：根据题目原文、建模分析和代码实现的产出，撰写完整论文。

执行步骤：

- 检查用户是否提供论文模板
- 基于 `references/latex-template/` 模板创建论文项目，每问一个独立 `.tex` 文件
- **每问先写方法对比**（>= 2 种方法），再详细展开选定方法的建立与求解
- **每张图 >= 100 字分析**，避免死板罗列
- 使用段落式自然陈述，遵循去 AI 味写作指南
- **附录规范**：代码用 lstlisting 原版粘贴，运行数据用 longtable
- **正文控制在 <= 25 页**

**产出文件**：

- `paper/` - LaTeX 项目目录（`main.tex` + 子文件 + `figures/` + `references.bib`）
- 编译产物：`paper/main.pdf` - 单个 PDF 论文

### 第四阶段：编译与交稿自查

**执行任务**：编译 LaTeX 项目，修复错误，执行交稿前自查清单。

**编译步骤**：

```bash
cd paper/
latexmk -xelatex main.tex
```

如果编译报错，分析错误日志（`main.log`），修复 `.tex` 文件后重新编译，循环直到零错误产出 `main.pdf`。

**交稿自查清单（必须逐项检查，不通过则修复后重新编译）**：

**排版与篇幅**
- 摘要恰好占满一页（不溢出第二页，也不留大片空白）
- 正文总计 <= 25 页（摘要到评价与推广）
- 各问篇幅大致均衡，不能出现某问 8 页另一问 2 页的失衡
- 图片无溢出页面边界，表格无截断

**内容硬伤**
- 摘要中包含每一问的具体数值结果（不能只有"效果良好"之类的定性结论）
- 问题重述是自己的话，不是原题的复制粘贴
- 符号说明表中的每个符号在正文中都被用到，正文中出现的符号都在表中有定义
- 每个模型假设在后续建模/求解中至少被引用一次（不能列了假设但从不使用）
- 正文中的数值结果与附录数据表格、代码实际输出三者一致，无矛盾
- 每张图的分析文字 >= 100 字，且描述的是图中可见的具体特征（趋势/拐点/峰值），不是泛泛而谈
- 模型评价的"缺点"部分有实质内容（不能写"数据量有限"之类的万能废话），要针对当前模型的真实局限

**去 AI 味**
- 全文搜索以下词汇并替换：值得注意的是、综上所述、本文创新性地、深入探讨、进一步地、不仅...而且、总而言之
- 无连续三个以上段落以相同句式开头
- 无表情符号和装饰性排版元素

**引用完整性**
- 所有 `\cite{}` 在 `references.bib` 中有对应条目
- 所有图表编号（图X、表X）在正文中被引用且顺序正确
- 附录代码是完整原版，未删减

## 使用建议

1. **按顺序执行各阶段**：建模分析 -> 代码实现 -> 论文撰写
2. **保持文件组织**：为每个建模项目创建独立文件夹
3. **术语一致性**：建模分析阶段创建的术语表格应被后续阶段使用
4. **鼓励创新**：算法资源库和决策指南仅供参考，不限制方法选择
