# 审核提示词（Review Prompts）

本文件收录两个**即插即用**的审核 prompt，面向数学建模竞赛交付前的最终自检。使用方法：把对应 prompt 整段复制粘贴给 Cascade（或任何 LLM 代理），Cascade 会依据 prompt 要求，对当前工作目录下的论文 / 代码 / 输出结果做系统性审核并给出修复方案。

**两个 prompt 的共同硬性要求**：

1. 必须**与题目原文交叉验证**：是否 100% 合题、是否完整解答、是否正确、是否可进一步优化
2. 必须列出**具体位置 + 具体问题 + 具体修复方案**，不得空话（如「结构需优化」「建议改进」）
3. 必须**基于实际读取到的文件内容**做结论，严禁无根据的空话

---

## Prompt A：论文审核（Paper Review）

复制以下整段 prompt 给 Cascade 即可启动论文审核。

```text
# 任务：数学建模论文交付前终审（与题目交叉验证 + 得奖论文对标）

你是一名严格的数模竞赛国奖评委兼论文手教练。请对当前工作目录下的论文项目
（`paper/` 源码 + `paper/main.pdf` 编译产物）执行系统性终审。审核必须完全基于
实际读取到的文件内容，给出具体位置和修复方案，严禁空话、套话、"建议优化"类
模糊结论。

## 第一步：输入采集（强制，不得跳过）

依次读取并在报告开头列出每个文件的字数 / 页数 / 行数等定量指标：

1. 比赛规则文档（`规则摘要.md` 或 `规则原文.md`）—— 页数上限、字号、摘要长度、
   参考文献格式、是否需要目录、附录政策
2. 题目原文 markdown（`X题.md`）—— 每一问的具体要求逐条列出
3. 论文 LaTeX 源码：`paper/main.tex`、`paper/abstract.tex`、`paper/question1.tex`
   ~ `paper/questionN.tex`、`paper/appendix.tex` 等所有子文件
4. 编译产物 `paper/main.pdf`（若存在，通过 pdf skill 逐页读取，统计实际页数、
   目录页数、摘要页数、每问页数、附录页数）
5. 代码产物 `output/q{n}/q{n}_*.csv` 的数值（用于与论文正文数值交叉核对）
6. 至少 5 份优秀得奖论文（从 `references/Outstanding Thesis/` 中取，要求
   覆盖不同年份 + 国赛 / 美赛混合），用 pdf skill 提取摘要页排版、目录结构、
   每问章节组织方式作为对标基准

## 第二步：合题性检查（100% 合题 / 完整解答 / 正确）

对照**题目原文**的每一问，用表格列出：

| 题目第 k 问 | 题目具体要求（原文摘录） | 论文中对应章节 | 论文给出的数值结果 | 代码 CSV 中对应数值 | 三者是否一致 | 缺漏项 |
|-----------|-----------------------|---------------|-------------------|-------------------|-------------|-------|

若任何一问有以下情况，**标记为严重问题**：
- 题目要求回答的子问题未在论文中出现
- 论文数值与 `output/q{n}/*.csv` 不一致
- 论文数值与附录数据不一致
- 题目指定的变量 / 参数 / 单位被遗漏或改动

## 第三步：与至少 5 份得奖论文对标

选取 5 份覆盖**不同年份、不同比赛**（国赛 2019/2020/2021/2022/2023、美赛任意年份）
的 O 奖 / 国一论文，对比以下维度并给出得奖论文的实际页数 / 结构数据：

| 维度 | 得奖论文 1 | 得奖论文 2 | 得奖论文 3 | 得奖论文 4 | 得奖论文 5 | 本文 | 差距 |
|-----|-----------|-----------|-----------|-----------|-----------|------|------|
| 正文总页数 | | | | | | | |
| 摘要页数 | | | | | | | |
| 目录页数 | | | | | | | |
| 每问平均页数 | | | | | | | |
| 每问图片数量 | | | | | | | |
| 附录内容构成 | | | | | | | |
| 摘要版式（三段 / 分栏 / vfill 等） | | | | | | | |
| 图片利用率（是否存在大片留白） | | | | | | | |

从对标中提炼出**本文与得奖论文的具体差距清单**，逐条给出修复方案。

## 第四步：用户实测反馈清单（逐条核对）

以下是用户在实际使用本 skill 后反馈的**具体痛点**，必须逐条核对并给出本论文
是否命中、命中位置、修复方案：

> 用户原话（不得稀释）：
> - 整体排版不太好，多处留白应该紧凑，图片应该足够清晰足够大利用好空间
> - 目录最多两页，最好压缩到一页
> - 摘要页可以参考示例 PDF 优秀得奖论文的排版
> - 部分表格溢出了左右
> - 部分正文内容是一坨屎，你没改，且不符合整体论文规范要求、准确科学的语言，
>   不要多说，大量数学推导和过程等
> - 引号 "" 有问题，中文引号不正常，左右的
> - 附录只应该放仨代码（即每问一段完整原版代码，且仅此而已），不要多加戏
> - 参考至少五个不同年份不同比赛的实际得奖论文，结合他们的格式和本比赛的要求
>   来进行更正

逐条输出：

| # | 用户痛点 | 本文是否命中 | 命中位置（文件 + 行号 / 页码） | 修复方案（具体到替换哪段 / 改哪个参数） |
|---|---------|-------------|---------------------------|--------------------------------------|
| 1 | 排版留白过多 / 图片太小 | | | |
| 2 | 目录超过 2 页 | | | |
| 3 | 摘要页排版粗糙 | | | |
| 4 | 表格左右溢出 | | | |
| 5 | 正文语言松散、缺推导 | | | |
| 6 | 中文引号错误 | | | |
| 7 | 附录堆了非代码杂项 | | | |

**引号检查的具体方法**：在 `paper/*.tex` 中搜索 `"` 和 `"`、``` `` ``` 和 `''`，
列出所有混用直引号 `"` 的位置。LaTeX 中文引号应用 `"..."` 或直接录入 Unicode
左右引号 `""`，严禁使用 ASCII 直引号 `"`。

**表格溢出的检查方法**：对每个 `\begin{tabular}` 估算列总宽度，与 `\textwidth`
对比；若超出，给出修复方案（`tabularx` / `\resizebox{\textwidth}{!}{...}` /
换用 `longtable` / 纵向转置）。

## 第五步：语言与数学推导深度检查

对照 `references/roles/论文手说明.md` §科学严谨与数学推导规范（第三部分正文强化）：

- 列出所有「可能 / 猜测 / 推测 / 大概 / 或许 / 似乎 / 应该（表推断）」出现的位置
- 列出每问正文的**字数统计**（不含图表、公式、代码），对照 3000 字下限
- 列出每问正文的**带编号公式数量**，对照下限 4~5 个
- 列出每问正文的**多行 `align` 推导数量**，对照下限 2 个
- 列出所有段首铺垫 / 段末总结 / 过渡水词 / 夸赞形容词 / 比喻句的具体位置
- 抽查每一小节是否「替换到其他题目仍成立」，若成立则标记为「算法百科，必须重写」

## 第六步：排版硬性检查（LaTeX 层面）

- [ ] 摘要页是否独占一页（无溢出、无大片空白）
- [ ] 目录页数 ≤ 2 页（理想 1 页），若超限给出 `tocdepth` / 字号 / 行距调整方案
- [ ] 正文总页数 ≤ 规则上限（默认 25 页）
- [ ] 所有章节 / 图 / 表 / 公式编号被 `\ref` 或 `\eqref` 引用
- [ ] 无章节标题孤悬页底（Widow / Orphan）
- [ ] 无任何一页「仅一张图 + 下方大片空白」
- [ ] 无表格左右溢出 `\textwidth`
- [ ] 图片 dpi ≥ 300，宽度合理（0.6 ~ 0.85\textwidth，充分利用版面）
- [ ] 附录仅含完整原版代码（每问一段，与 `q{n}.py` 一一对应），无画蛇添足的
      「运行说明」「模型总结」「致谢」等与代码无关的内容
- [ ] 所有中文引号为正常左右引号（`"..."` 或 `""`），无 ASCII 直引号 `"`
- [ ] 所有破折号为中文破折号 `——`（而非 `--` / `-`）
- [ ] 参考文献格式符合规则（GB/T 7714 或比赛指定格式）

## 第七步：可优化点（正确性之外）

- 图片数量是否饱和（每问 4~8 张为常规，简单题下限 3 张）
- 灵敏度分析是否横向 ≥ 5 列
- 方法对比表是否 ≥ 5 列
- 主结果表是否 ≥ 4 列
- 表格是否使用 `booktabs` 三线表，无竖线
- 参考文献是否 ≥ 5 条且与正文 `\cite` 匹配
- 符号说明表是否与正文符号 1:1 对应

## 输出格式（必须遵守）

报告分以下六节：

1. **输入清单**：第一步读取到的文件定量指标
2. **合题性矩阵**：第二步的表
3. **得奖论文对标**：第三步的表 + 差距清单
4. **用户痛点核对**：第四步的表
5. **语言 / 推导 / 排版问题清单**：第五 + 六步，按文件 + 行号排序
6. **修复行动项**：第 1~5 步发现的所有问题，按优先级（阻塞 / 严重 / 一般）
   排序，每项包含：具体文件、具体行号 / 页码、原文摘录、修改后内容、
   预计影响页数

严禁输出「整体结构合理」「建议加强数学推导」「注意语言规范」这类模糊结论，
所有结论必须可落地到某个具体 `.tex` 文件的某一行。
```

## Prompt B：代码与运行结果审核（Code & Output Review）

复制以下整段 prompt 给 Cascade 即可启动代码审核。

```text
# 任务：数学建模代码与运行结果终审（与题目交叉验证）

你是一名严格的数模竞赛国奖评委兼编程手教练。请对当前工作目录下的代码项目
（`q1.py ~ qN.py` + `output/` 产物 + `data/` 输入）执行系统性终审。审核必须完全
基于实际读取到的文件内容，给出具体位置和修复方案，严禁空话、套话。

## 第一步：输入采集（强制，不得跳过）

依次读取并在报告开头列出每个文件的行数 / 字节数 / 产出文件列表：

1. 题目原文（`X题.md`）—— 逐条列出每一问的具体输入 / 输出要求、指定变量名、
   指定数值范围、指定单位
2. 附件清单（`附件清单.md`）—— 每个 `data/` 下文件的列名、行数、数据类型
3. 代码主脚本 `q1.py ~ qN.py`—— 逐文件读取并统计：
   - 随机种子是否固定
   - 是否使用相对路径
   - 是否在顶部 `CONFIG = {...}` 集中超参
   - 是否存在硬编码结论（`# 方法A更优` / `optimal_method = "A"` 这类）
   - 是否存在硬编码数值（从 CSV 应当读取、却直接写死在代码里的常量）
4. 代码输出 `output/q{n}/`—— 逐目录读取：
   - `q{n}_*.csv` 文件列表 + 每个 CSV 的列名 + 行数
   - `q{n}_log.txt`—— 是否存在 ERROR / WARNING / NaN / Inf
   - `q{n}_summary.json`（若存在）—— 关键数值指标
   - `figures/` 下图片数量 + 每张图的 dpi（从文件名或 PNG 元数据）
5. `README.md`—— 复现说明是否完整

## 第二步：合题性检查（100% 合题 / 完整解答 / 正确）

对照**题目原文**的每一问，用表格列出：

| 题目第 k 问 | 题目要求的输入（附件名 + 字段） | 代码实际读入的输入 | 题目要求的输出（变量名 + 单位 + 范围） | 代码实际输出（CSV 字段） | 三者是否一致 | 缺漏 / 偏差 |
|-----------|--------------------------------|-------------------|-----------------------------------------|------------------------|-------------|------------|

若任何一问有以下情况，**标记为严重问题**：

- 题目要求读入的附件字段被遗漏
- 题目要求输出的指标未在 CSV 中出现
- 输出的数值单位与题目要求不一致
- 输出的数值超出题目给定的物理 / 工程合理范围
- 题目要求的子问题未被单独求解（如「问题三第 2 小问」没有对应 CSV）

## 第三步：正确性检查（数值 / 约束 / 边界）

对每一问的主要产出，逐项核对：

| 检查项 | 方法 | 本问实际结果 | 判定 |
|-------|-----|-------------|------|
| 量级检查 | 对比同类问题的典型量级 | | |
| 约束满足 | 把最优解代入所有约束，计算松弛量 | | |
| 边界测试 | 输入数据取极端值，检查输出是否仍合理 | | |
| 基线对比 | 与最简单启发式（如 greedy / 平均分配）对比 | | |
| 交叉验证 | 预测 / 评价类模型至少做 5 折 CV | | |
| 数值稳定性 | 两次独立运行（相同 seed）结果是否完全一致 | | |
| 单位一致性 | 公式两侧量纲是否齐次 | | |

## 第四步：可复现性检查

- [ ] 所有 `np.random.seed` / `random.seed` / `torch.manual_seed` 固定到同一值
- [ ] 所有路径为相对路径（`Path(__file__).parent / "data" / "xxx.csv"`）
- [ ] 脚本能从**空的 `output/` 目录**一键跑通（只执行 `python q{n}.py`）
- [ ] 超参集中在顶部 `CONFIG = {...}` 块，未散落
- [ ] `requirements.txt`（或等价）列出所有依赖及版本

## 第五步：代码职责边界检查（编程手纯粹原则）

对照 SKILL.md §2.3「代码职责（纯粹原则）」，列出以下反模式的具体位置：

- [ ] `print` / 注释 / 输出文件中包含结论性判断（如「方法A更优」「本方法优于XX」）
- [ ] 变量名 / 函数名暗含结论（如 `optimal_method`、`best_algorithm`）
- [ ] 代码自动生成 `summary.md` / `对比报告.md` / 论文片段
- [ ] 代码硬编码了从论文反推的数字（应当从数据计算，却写死常量）
- [ ] 不同问的 CSV / 图文件互相覆盖（未按 `output/q{n}/` 分子目录隔离）

## 第六步：图表质量检查

| 问 | 图片数量 | 是否达到下限（常规 4~8 / 下限 3） | 命名是否为 `q{n}_fig{k}_*.png` | dpi 是否 ≥ 300 | 是否有把表格 / 规则画成图的反模式 |
|---|---------|----------------------------------|------------------------------|--------------|-------------------------------|

**反模式标记**：`plot_rule_summary` / `plot_schedule_table` 这类把文字表格塞进
matplotlib 图里的做法，应改为直接输出 CSV 交给论文手渲染 LaTeX 三线表。

## 第七步：目录结构检查

对照 SKILL.md §2.2 的标准目录结构，列出以下违反项的具体位置：

- [ ] `output/` 根目录平铺 `q1_*.csv`、`q2_*.csv` 未按问分子目录
- [ ] `figures/` 单独一级（应在 `output/q{n}/figures/` 下）
- [ ] 不同问的图文件名混在一起（无 `q{n}_` 前缀）
- [ ] 每问缺失自己的 `q{n}_log.txt` 或 `figures/` 子目录

## 第八步：可优化点（正确性之外）

- 算法效率：能否用向量化 / 矩阵运算替代 Python 循环
- 内存占用：大数据集是否需要分块读取
- 并行化：独立子问题是否可用 `multiprocessing` / `joblib` 加速
- 数值稳定：是否使用 `log-sum-exp` / 条件数较好的分解
- 可视化增强：关键图是否加入置信区间 / 误差棒 / 多方法对比
- 代码风格：是否存在超过 100 行的函数需要拆分
- 日志完备：是否对每一步关键计算都有 `print` 标签 + 数值

## 输出格式（必须遵守）

报告分以下七节：

1. **输入清单**：第一步读取到的文件定量指标
2. **合题性矩阵**：第二步的表
3. **正确性矩阵**：第三步的表
4. **可复现 / 职责 / 结构违反项**：第四 + 五 + 七步
5. **图表质量表**：第六步
6. **可优化点清单**：第八步
7. **修复行动项**：第 1~6 步发现的所有问题，按优先级（阻塞 / 严重 / 一般）
   排序，每项包含：具体文件、具体行号、原代码摘录、修改后代码、
   预计影响（正确性 / 可复现性 / 性能）

严禁输出「代码整体良好」「建议优化性能」「注意可复现性」这类模糊结论，
所有结论必须可落地到某个具体 `.py` 文件的某一行、或某个具体 CSV 的某一列。
```

## 使用建议

1. **阶段四自查时顺序执行**：先 Prompt B 审代码（确保数值基础正确），再 Prompt A 审论文（因为论文数值引用代码产物）
2. **每次修改后重跑**：修完一轮问题后，重新粘贴对应 prompt 做第二轮审核，直到报告无阻塞 / 严重项
3. **prompt 可按题目定制**：若题目有特殊要求（如「必须用特定算法」「必须在特定时间内求解」），在第二步「合题性检查」表格中追加该列
4. **对标论文不够时**：若 `references/Outstanding Thesis/` 下不足 5 份得奖论文，Cascade 应主动索要或用 `paper_search` 工具补充
