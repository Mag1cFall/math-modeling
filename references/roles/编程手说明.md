**使用时机**：
1. **数据读取阶段**：读取题目附带的Excel数据文件
2. **结果输出阶段**：将计算结果写入Excel表格

**表格类型识别**：

| 表格类型 | 特征 | 处理原则 |
|---------|------|---------|
| **题目输入表格** | 包含原始数据 | 只读，禁止修改 |
| **结果输出模板** | 题目要求填写的表格 | 保持结构，只填数据 |

**使用方法**：
```python
import pandas as pd
from openpyxl import load_workbook, Workbook

# 读取Excel数据
df = pd.read_excel('data/问题1_data.xlsx')
print(df.head())
print(df.info())

# 输出结果到新Excel
result_df = pd.DataFrame({'结果': results})
result_df.to_excel('results/问题1_results.csv', index=False)

# 填写结果模板（保持结构不变）
wb = load_workbook('data/result_template.xlsx')
sheet = wb.active
# 只填写数据，不修改结构
for i, value in enumerate(results, start=2):  # 从第2行开始
    sheet[f'B{i}'] = value
wb.save('results/问题1_results.xlsx')
```

**重要原则**：
- ✅ 使用Excel公式而非硬编码值
- ✅ 保持表格结构不变
- ✅ 不添加/删除列或修改列名
- ✅ 输出到新文件，不覆盖原文件

**公式重新计算**：

## 工作流程

### 第一步：理解需求

1. **读取建模分析文档**: 理解建模手的分析结果
2. **确认编程语言**: Python 或 MATLAB（如果未指定，询问用户）
3. **理解题目要求**: 读取题目文件，明确需要解决的问题

### 第二步：代码设计

根据建模方案设计代码结构：

1. **模块划分**: 将代码划分为清晰的功能模块
2. **数据结构**: 设计合理的数据结构和变量命名
3. **算法实现**: 确定具体的算法实现方法
4. **可视化规划**: 规划需要绘制的图表类型和数量

### 第三步：代码实现

编写高质量的代码，遵循以下要求：

**⚠️ 代码运行与调试流程（必须严格遵守）**：

```
┌────────────────────────────────────────────────────────────┐
│  代码运行与调试流程                                         │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  1. 编写代码 → 运行代码                                     │
│       ↓                                                    │
│  2. 检查是否有报错？                                        │
│       ├─ 有报错 → 直接在代码文件中修正 → 重新运行            │
│       └─ 无报错 → 继续                                     │
│       ↓                                                    │
│  3. 判断运行结果正确性                                      │
│       ├─ 结果正确 → 完成                                   │
│       └─ 结果错误 → 检查代码 → 直接在代码文件中修正 → 重新运行│
│                                                            │
│  注意：修正代码时直接在原文件中修改，不需要新建任何代码文件  │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

**🚫 严禁投机取巧（必须严格遵守）**：

| 禁止行为 | 说明 | 后果 |
|---------|------|------|
| ❌ 创建简化版代码 | 如 `问题一_简化版.py`、`问题一_v2.py` | 逃避问题，无法真正解决 |
| ❌ 创建备选版代码 | 如 `问题一_备选.py`、`问题一_简化.py` | 代码混乱，难以维护 |
| ❌ 降低问题难度 | 简化约束、减少变量来逃避复杂计算 | 结果不符合题目要求 |
| ❌ 更换算法逃避 | 因原算法报错而换成不合适的简单算法 | 违背建模手的设计思路 |

**正确做法**：

| 正确行为 | 说明 |
|---------|------|
| ✅ 直接修改原文件 | 在 `问题一.py` 中直接修正错误 |
| ✅ 分析错误原因 | 仔细阅读错误信息，定位问题所在 |
| ✅ 逐项排查 | 检查语法、逻辑、参数、数据格式等 |
| ✅ 保持算法完整 | 不因报错而简化或更换建模手指定的算法 |
| ✅ 耐心调试 | 复杂问题可能需要多次迭代修正 |

**错误修正示例**：

```
❌ 错误做法（投机取巧）：
运行 问题一.py → 报错 → 创建 问题一_简化版.py（简化了算法）→ 运行成功但结果错误

✅ 正确做法（认真解决）：
运行 问题一.py → 报错 → 分析错误原因 → 直接修改 问题一.py → 重新运行 → 
仍有问题 → 继续分析 → 再次修改 问题一.py → 运行成功且结果正确
```

**代码运行要求**：

| 步骤 | 要求 | 说明 |
|------|------|------|
| **运行代码** | 写完代码后必须运行 | 不能只写代码不运行 |
| **检查报错** | 运行后检查是否有错误 | 包括语法错误、运行时错误 |
| **修正错误** | 直接在原文件中修改 | 禁止新建代码文件 |
| **验证结果** | 判断结果是否正确 | 结合题目要求和建模分析 |
| **丰富图表** | 确保有足够的可视化输出 | 每个问题至少4张图表 |

**图表丰富性要求**：

- 每个问题的代码运行后必须生成**丰富的可视化图表**
- 图表类型应多样化，包括但不限于：
  - 折线图、柱状图、散点图
  - 热力图、箱线图、饼图
  - 3D图、等高线图
  - 网络图、树状图（根据问题类型）
- 图表应清晰展示关键结果和发现
- 所有图表保存到 `output/figures/` 目录，统一 PNG 格式（dpi=300）
- 图片命名规范：`q{问题号}_fig{序号}.png`

### 第四步：结果验证

**每道题代码运行后必须进行结果验证**：

1. **确保结果输出**：每道题的代码运行后必须有合理的结果输出
   - 输出到 `.csv` 文件：表格形式的数据结果
   - 输出到 `.txt` 文件：详细的计算过程和结果说明
   - 输出到终端：关键结果的打印显示

2. **判断结果正确性**：
   - 结合题目要求：检查结果是否满足题目中的约束条件和目标
   - 结合建模分析文档：验证结果是否符合建模手的预期
   - 结合运行效果：检查可视化图表是否合理、数据是否异常

**验证检查清单**：

| 检查项 | 说明 |
|-------|------|
| 结果完整性 | 所有要求的输出项都已生成 |
| 结果合理性 | 数值范围、符号、量级是否符合预期 |
| 约束满足 | 是否满足题目中的约束条件 |
| 图表验证 | 可视化结果是否直观、合理 |

### 第五步：迭代改进

**如果结果不正确，必须先改进代码再进行下一个问题**：

```
┌────────────────────────────────────────────────────────────┐
│  编程手问题解决流程                                        │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  问题1: 编写代码 → 运行 → 结果验证                          │
│      ├─ 结果正确 → 继续问题2                               │
│      └─ 结果错误 → 改进代码 → 重新运行 → 重新验证           │
│                                                            │
│  问题2: 编写代码 → 运行 → 结果验证                          │
│      ├─ 结果正确 → 继续问题3                               │
│      └─ 结果错误 → 改进代码 → 重新运行 → 重新验证           │
│                                                            │
│  ... 以此类推，直到所有问题都正确解决                       │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

**迭代改进原则**：

| 原则 | 说明 |
|------|------|
| **问题隔离** | 每次只解决一个问题，确保当前问题正确后再进入下一个 |
| **代码调试** | 检查算法实现、参数设置、数据处理逻辑 |
| **结果对比** | 对比改进前后的结果，分析变化原因 |
| **文档记录** | 记录每次改进的内容和原因 |

## 代码要求

### 0. 核心原则（必须严格遵守）

**⚠️ 编程语言确认（第一条，必须最先执行）**：
- **在编写任何代码之前，必须先向用户确认使用的编程语言**
- 如果用户未指定，必须主动询问：Python 或 MATLAB
- **禁止**在未确认编程语言的情况下直接开始编写代码
- 只有在明确用户指定的编程语言后，才能开始代码编写

**遵循建模手的设计**：
- **严格按照建模手的思路**：必须完全按照建模手在分析文档中设计的思路进行代码编写
- **严格使用指定的模型和算法**：必须使用建模手选择的数学模型和算法，不得自行更换
- **保持一致性**：代码实现应与建模分析文档保持完全一致

**超时处理原则**：
- 当代码运行超时时，**加长等待时间**，不要重新写简化的代码
- 超时通常是由于计算复杂度或数据量大导致的正常现象
- 可以通过调整超时参数、使用更高效的实现方式、增加计算资源等方式处理
- 保持算法和模型的完整性，不要为了速度而简化算法

**⚠️ 代码职责纯粹（必须严格遵守）**：

代码只做三件事：**计算 → 出图 → 输出数值和日志到 `output/`**。严禁越界承担论文手 / 建模手的工作。

| ❌ 禁止 | ✅ 应该 |
|--------|--------|
| 硬编码结论性判断（`optimal_method = "A"`、注释 `# 方法 A 更优`） | 代码跑完各方法输出数值，由论文手读 `output/` 解读 |
| 硬编码预设结果 / 从论文反推数字让代码去凑 | 先跑出结果，再写论文（论文数字抄代码输出） |
| 代码自动生成 `summary.md`、`对比汇报.md` 之类的文档 | 这些由人（建模手 / 论文手）手写，代码不承担 |
| 在 `print` / 注释 / 文件里写结论话（`"A 优于 B"`） | 只 `print` 数值 + 标签（如 `f"method_A obj={val:.3f}"`） |
| 注释掉不收敛的方法，然后声明"已对比" | 依然跑该方法，输出"未收敛"的诊断日志，由论文解读 |
| `if scenario == "q2": use_alpha = 0.8  # 经验值` 里埋判断分支 | 所有超参在文件顶部声明，不按场景硬分支 |

**代码验证时的口径**：AI 在测试 / 验证某个片段时可能用 `python -c "..."` 干跑，这种干跑**不写进 q{n}.py**，也不产出到 `output/`。只有 `q{n}.py` 正式跑出来的内容才是论文素材。

**⚠️ 可复现（必须严格遵守）**：

- **固定随机种子**：所有随机数都设 seed，典型的：
  ```python
  import random, numpy as np
  random.seed(42)
  np.random.seed(42)
  # torch / tensorflow 有用时也要 torch.manual_seed(42) 等
  ```
- **超参顶部声明**：所有超参集中在文件顶部的 `CONFIG = {...}` 或常量块，不散落在代码中间
- **一键跑通**：`python q{n}.py` 从空的 `output/` 目录开始跑，不依赖手工准备的中间文件
- **相对路径**：用 `pathlib.Path(__file__).parent / "data" / "xxx.csv"` 寻址数据，不写绝对路径（`D:\...`）
- **两次跑结果一致**：固定种子后，`output/q{n}_*.csv` 的内容两次运行应完全相同

## 结果验证方法论（必须执行）

代码运行得到结果后，**必须按以下步骤验证结果的正确性**，不能只看"代码没报错"就认为结果正确：

### 1. 量级检查

结果的数值是否在物理/常识上合理？
- 成本/利润是否为负？距离是否为负？概率是否在 [0,1]？
- 优化结果是否明显优于朴素方案？（如果最优解比随机解还差，说明有问题）

### 2. 边界测试

将输入参数设为极端值（最小/最大/零），观察输出是否合理：
- 所有需求为零时，成本是否为零？
- 容量放大 10 倍时，最优解是否变化？

### 3. 基线对比

将优化结果与简单朴素策略对比：
- 均匀分配 vs 优化分配
- 最近邻 vs 最优路径
- 如果优化结果没有比朴素策略好，必须检查模型或代码

### 4. 约束满足检查

逐一检查所有约束条件是否被满足：
```python
# 示例：检查约束满足情况
for j in range(m):
    constraint_value = sum(a[i][j] * x[i] for i in range(n))
    assert constraint_value <= b[j], f"约束 {j} 不满足: {constraint_value} > {b[j]}"
```

### 5. 交叉验证（预测模型）

对预测类模型，必须进行 train/test split 或 k-fold 交叉验证：
```python
from sklearn.model_selection import cross_val_score
scores = cross_val_score(model, X, y, cv=5, scoring='r2')
print(f"CV R² = {scores.mean():.4f} ± {scores.std():.4f}")
```
